Recursion — Deep Dive Full Guide
A complete visual guide to understanding recursion from scratch — how it works, how the call stack behaves, patterns, pitfalls, and problems to practice.
Introduction
What is Recursion?

📌 Definition

Recursion is when a function calls itself from within its own body. Instead of solving the whole problem at once, you break it into smaller instances of the same problem — until you hit a stopping condition (the base case).

🎯 Why Use It?

Many problems are naturally recursive — trees, graphs, divide-and-conquer, backtracking. Recursion often produces cleaner, more readable code than equivalent iterative solutions. It's the foundation for QuickSort, MergeSort, DFS, and BFS.

Rule 1 — Base Case

Every recursive function must have a base case — a condition where it stops calling itself. Without it: infinite recursion → StackOverflow.

Rule 2 — Progress

Each recursive call must move closer to the base case. If your input doesn't change, you'll never stop.

Mechanics
How Recursion Actually Works
📞

Call Phase

The function is called. A new stack frame is pushed onto the call stack, holding local variables and the return address.

📥

Stack Builds Up

Each call adds a frame. The stack keeps growing until the base case is reached and no more calls are made.

📤

Return Phase (Unwinding)

Once the base case returns, frames are popped one by one. Each frame uses the returned value to complete its own computation.

Memory Note: Each stack frame takes memory. For very deep recursion (e.g., n = 100,000), you risk a StackOverflowError. Consider converting to iterative or using tail recursion.

The Base Case — Your Exit Condition

✓ With Base Case

if (n == 0) return 1; return n * factorial(n-1);

Stops at n=0. Stack unwinds correctly. Program completes with the right answer.

✗ Without Base Case

return n * factorial(n-1); // calls forever...

Never stops. Stack overflows. JVM throws StackOverflowError.

Common Base Case Patterns
🔢
Numeric Boundary
if (n == 0) return 1;  if (n <= 1) return n;
📋
Empty Collection
if (arr.length == 0) return;  if (list.isEmpty()) return 0;
🌳
Null Node (Trees)
if (node == null) return;
📏
Index Bounds
if (left > right) return -1;  (Binary Search)
💡 Pro Tip: Always write the base case first before writing the recursive step. It forces you to think about when the problem is small enough to solve directly.
Fibonacci — Two Recursive CallsJAVA
int fib(int n) { if (n <= 1) return n; // base cases: fib(0)=0, fib(1)=1 return fib(n - 1) + fib(n - 2); // two recursive calls! } // fib(0)=0 fib(1)=1 fib(2)=1 fib(3)=2 fib(4)=3 fib(5)=5
Call Tree for fib(5)
Problem: Naive Fibonacci recalculates the same subproblems many times. For fib(5), fib(2) is computed 3 times. Time complexity is O(2ⁿ) — exponential. The fix is Memoization (Tab 4).

Memoization — Cache Your Results

✗ Naive O(2ⁿ)

int fib(int n) { if (n <= 1) return n; return fib(n-1) + fib(n-2); } // recalculates everything

✓ With Memo O(n)

int[] memo = new int[N+1]; Arrays.fill(memo, -1); int fib(int n) { if (n <= 1) return n; if (memo[n] != -1) return memo[n]; memo[n] = fib(n-1) + fib(n-2); return memo[n]; }
General Memoization Pattern
Map<Integer,Integer> memo = new HashMap<>(); int solve(int state) { if (isBaseCase(state)) return baseValue; // 1. base case if (memo.containsKey(state)) // 2. check cache return memo.get(state); int result = solve(smallerState); // 3. recurse memo.put(state, result); // 4. cache result return result; }
When to use Memoization: Any time you notice your recursive function is called with the same arguments more than once. Classic cases: Fibonacci, Coin Change, Knapsack, Longest Common Subsequence. This is the foundation of Dynamic Programming.
Pro Tips for Writing Recursive Code
1
Write the Base Case First

Before anything else, ask: "When is the answer trivially known?" Write that condition first. It anchors your thinking and prevents infinite recursion bugs.

2
Trust the Recursion ("Leap of Faith")

Assume that f(n-1) already works correctly. Don't try to trace through every call in your head — just define how to build the answer for n using the answer for n-1.

3
Verify Your Input Shrinks

Every recursive call must pass a smaller or simpler version of the input. If not, you'll never reach the base case.

4
Draw the Call Tree

For any recursion you don't understand, draw it. Write the function call at the top, then draw branches for each recursive call. Trace the return values back up. This reveals overlapping subproblems too.

5
Add Memoization When You See Overlap

If your call tree has the same node appearing more than once, you need caching. Add a HashMap or array and check it before computing.

6
Know When to Go Iterative

For very deep recursion (n > 10,000+), the stack can overflow. Consider converting to bottom-up DP or an explicit stack. Java's default stack is typically ~500–1000 deep calls.
